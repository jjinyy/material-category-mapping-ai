# 자재 분류 파이프라인 구조 정리

## 1. 전체 흐름 (한 줄 요약)

```
자재명 입력 → 전처리·언어감지 → [1] FAISS로 상위 50개 후보 + 점수 → [2] 규칙 적용(apply_rule) → [3] 점수로 정렬 → 최종 목록
```

---

## 2. 상세 설명 (규칙이 언제·어떻게 적용되는지)

### 2-1. 1단계: ML 후보 만들기

- 사용자가 **자재 타입(ROH1/ROH2)** 과 **자재명**(예: 메밀, 옥수수 침지액)을 입력합니다.
- FAISS가 **전체 카테고리** 중 유사도 상위 **50개**를 뽑고, 각각 **Score = 1/(1+거리)** 로 점수를 줍니다.
- 그 다음 **선택한 자재 타입과 다른 TYPE** 은 제거합니다.  
  → 예: ROH1 선택 시 50개 중 ROH1만 남기므로, **화면에 넘어가는 개수는 매번 다를 수 있습니다.**

이 시점의 리스트를 **“ML 결과”** 라고 부릅니다.  
예: 메밀 입력 시 ML 결과 = [ (원맥&밀, 5.52), (메밀가루, 4.52), (메밀, 4.19), … ]

---

### 2-2. 2단계: 규칙 매칭 여부 확인

- **rule_base.json** 을 위에서부터 읽어서, **자재명에 키워드가 포함**되면 그 규칙 하나를 골라 **matched_code** 를 정합니다.  
  (예: "메밀" 입력 → "메밀" 키워드 규칙 매칭 → code = AAMG0030)
- **어떤 규칙에도 안 걸리면** → 규칙 적용 없이 ML 결과를 그대로 쓰고, 3단계(정렬)로 갑니다.

---

### 2-3. 규칙이 매칭됐을 때만 나뉘는 두 가지 경우

규칙이 매칭되면 **“이 카테고리( matched_code )를 1등으로 보여주자”** 가 목표이고,  
**점수는 “ML 1등 점수보다 살짝만 높게”** 로 통일합니다.

**케이스 A: 규칙 코드가 ML 결과 “안에 없음”**  
(예: "csl" → 옥수수 침지액 FUPA0090 인데, ML 50개 안에 FUPA0090이 없음)

- 규칙 카테고리 **한 줄을 새로 만들어** 리스트에 넣습니다.
- 이 줄의 점수 = **현재 ML 1등 점수 + 0.01** (상한 0.97).  
  → ML 1등이 6.43%면 규칙 행은 7.43%로 표시되어, 정렬 후 **1등**이 됨.
- 즉, **“ML에는 안 나오지만 규칙으로 추가해서 1등으로 노출”** 되는 경우입니다.

**케이스 B: 규칙 코드가 ML 결과 “안에 있음”**  
(예: "메밀" → AAMG0030(메밀) 이 ML 결과 3등으로 있음)

- ML 결과에서 **그 코드 행을 빼고**, 같은 카테고리 정보로 **새 행 하나**를 만듭니다.
- 이 새 행의 점수 = **현재 ML 1등 점수 + 0.01** (상한 0.97).  
  → ML 1등이 5.52%면 규칙 행은 5.53%로, 정렬 후 **1등**이 됨.
- 나머지(원맥&밀, 메밀가루 등)는 원래 ML 점수 그대로 두고, 규칙 행 **뒤에** 이어붙입니다.

공통점:  
- **규칙이 매칭되면 → 그 카테고리는 항상 1등으로 노출.**  
- **점수는 “ML 1등보다 살짝만 높은 값” 하나로 통일** (바닥 70% 같은 건 없음).

---

### 2-4. 3단계: 최종 정렬

- **규칙 적용 후 리스트**를 **Score 기준 내림차순**으로 한 번 더 정렬합니다.
- 규칙으로 준 점수 = (ML 1등 + 0.01) 이므로, **규칙 매칭된 행이 1등**, 그 다음이 ML 1등, 2등, … 순서가 됩니다.

---

### 2-5. 정리 표

| 상황 | 규칙 적용 | 1등 | 규칙 행 점수 |
|------|-----------|-----|--------------|
| 규칙 매칭 안 됨 | 없음 | ML 1등 그대로 | - |
| 규칙 매칭됨, 규칙 코드가 **ML 결과에 없음** | 지지선 추가 | 규칙 카테고리 | ML 1등 + 0.01 |
| 규칙 매칭됨, 규칙 코드가 **ML 결과에 있음** | 1등으로 올림 | 규칙 카테고리 | ML 1등 + 0.01 |

---

## 3. Rule Base (data/rule_base.json)

- **구조**: JSON 배열. 각 항목은 `keywords`, `code`, `type`, (선택) `regex`, `priority` 등.
- **실제 사용**: `main.py`에서는 **keywords만** 사용함.  
  - 한글: 자재명에 키워드 **문자열 포함** 여부로 매칭.  
  - 영문/숫자: 단어 단위 또는 포함 여부로 매칭.
- **순서**: 파일 안에서 **위에 있는 규칙이 우선**.  
  - 예: "필름" 규칙이 "리드필름" 규칙보다 위에 있으면, "리드필름" 입력 시에도 "필름"에 먼저 걸려서 PLMF0030(일반 필름)이 1등이 됨.

---

## 4. 리드필름이 다른 필름으로 나오던 이유 (참고)

- 카테고리에는 **리드필름 전용** 코드가 있음:
  - **PLMF0050**: 리드필름-일반 (Lid Film)
  - **PLMF0060**: 리드필름-햇반 (Lid Film-Hetbahn)
- 그런데 rule_base에는 **일반 "필름"** 규칙만 있음:
  - 키워드: `["film", "필름", ...]` → code **PLMF0030** (일반 필름)
- "리드필름"에는 **"필름"**이 포함되어 있어서, **일반 필름 규칙에 먼저 걸림**.
- 그래서 1등이 PLMF0030(일반 필름), 점수 85%(RULE_FLOOR)로 나오고,  
  리드필름 전용(PLMF0050/PLMF0060)은 ML 결과에만 있을 수 있어 2등 이하로 밀림.

**해결**:  
- **리드필름**·**햇반리드** 등 **더 구체적인 키워드** 규칙을 추가하고,  
- 이 규칙들을 **일반 "필름" 규칙보다 JSON에서 위(앞)**에 두면,  
  "리드필름" 입력 시 리드필름 전용 코드가 1등으로 적용됨.

---

## 5. 설정값 (config.py)

| 항목 | 값 | 설명 |
|------|-----|------|
| CLASSIFICATION_TOP_N | 50 | FAISS에서 가져오는 후보 개수 (타입 필터 후 개수는 입력마다 다름) |
| RULE_APPLY_ONLY_IF_IN_ML | True | False면 규칙 코드가 ML에 없어도 예전처럼 별도 점수로 넣는 방식 (현재는 True로 “ML에 없으면 지지선 추가”만 사용) |
| RULE_FALLBACK_MARGIN | 0.01 | 규칙으로 1등 줄 때 쓰는 점수 = ML 1등 + 이 값 (화면 1% 차이) |
| RULE_CAP | 0.97 | 규칙 점수 상한 (0.97 = 97%) |

---

## 6. 요약

- **규칙이 매칭되면** → 그 카테고리는 **항상 1등**, 점수는 **ML 1등 + 0.01** 로 통일.
- **규칙이 매칭 안 되면** → ML 결과만 그대로 사용.
- **rule_base.json** 에서는 **먼저 나오는 규칙이 우선**이므로, 구체적인 키워드(리드필름, 메밀 등)를 일반 키워드(필름, 곡물 등)보다 **앞**에 두는 것이 좋음.
